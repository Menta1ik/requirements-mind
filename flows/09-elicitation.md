# Сценарий: 09-elicitation (Сбор требований с нуля / Elicitation Mode)

Этот сценарий описывает интерактивный процесс сбора требований с нуля, когда у аналитика есть только сырая идея и нет готовых входных материалов. Сценарий основан на гибридном стандарте **«Практичный аналитик»** (JTBD + Use Cases + ISO 29148).

---

## 👥 Участвующие роли
* **A3 — Elicitation Expert** (ИИ-Интервьюер требований)
* **Onboarding Wizard (A0)** (ИИ-Ассистент старта проекта)
* **Системный аналитик** (Пользователь)

---

## 🏁 Входные данные
* Сырая бизнес-идея или краткое название проекта, переданное аналитиком в чате IDE.
* Пустая папка `input/` или наличие только базовых шаблонов.

---

## ⚙️ Пошаговый процесс

### 🌟 Шаг 1: Активация режима Elicitation
1. Аналитик инициирует сессию с ИИ-ассистентом:
   > **«Мэри, давай начнем сбор требований для нового проекта delivery-app»**
2. ИИ-ассистент проверяет состояние проекта в `state.json` (или создает структуру папок, если проект абсолютно новый).
3. ИИ-ассистент перенаправляет сессию на навык **`a3-elicitation`** и устанавливает `profile: 6` в `state.json`.

---

### 💬 Шаг 2: Интерактивный диалог (Сбор требований)
1. **Фаза JTBD (Бизнес-ценность):**
   * ИИ-агент `a3-elicitation` задает вопросы о целевой аудитории и формулирует гипотезы Jobs-to-be-Done:
     * *«Когда я хочу заказать еду, я хочу видеть время доставки, чтобы я мог спланировать обед»*.
   * Аналитик подтверждает или корректирует формулировки.
2. **Фаза Use Cases (Функционал):**
   * ИИ-агент опрашивает аналитика о границах системы и ключевых пользовательских сценариях.
   * Выявляются основные функциональные блоки (например, Каталог, Корзина, Оплата, Личный кабинет).
3. **Фаза ISO 29148 NFR (Системные требования):**
   * ИИ-агент опрашивает аналитика о нефункциональных требованиях: безопасность (авторизация, шифрование), производительность (RPS, время отклика) и интеграции с внешними API.

---

### 📝 Шаг 3: Автогенерация требований на диске
Как только сессия завершается (все ключевые аспекты оговорены), ИИ-агент:
1. Записывает структурированный файл **`projects/<project-name>/input/requirements.md`** со всеми деталями (включая JTBD, функциональные блоки, Use Cases, NFR).
2. Записывает первичный стартовый скелет **`projects/<project-name>/context.md`** со ссылками на `input/requirements.md`.
3. Устанавливает статус проекта `status: "onboarding"` с `profile: 6` и `active_agents: ["a0", "a3"]` в `state.json`.

---

### 🏁 Шаг 4: Фиксация и переход к разработке спецификации
После того как ИИ сгенерировал файлы, ИИ-агент автоматически предлагает запустить команду фиксации онбординга в терминале (аналитику достаточно просто подтвердить запуск в чате):

```bash
# ЗАПУСКАЕТСЯ АВТОМАТИЧЕСКИ ИИ-АГЕНТОМ ПОСЛЕ ПОДТВЕРЖДЕНИЯ В ЧАТЕ
uv run cli.py onboard --project=<project-name>
```

**Что происходит:**
* CLI проверяет наличие `input/requirements.md` и `context.md`.
* CLI переводит статус проекта в **`intake`**, приглашая запустить глубокий Intake-анализ с помощью `a1-intake-analyst` для перехода к созданию BRD.
