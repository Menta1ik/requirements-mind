**Requirements Mind** — это CLI-first, markdown-first и multi-agent инструмент для автоматизации работы с проектными требованиями, разработки спецификаций и проведения системного и бизнес-анализа на базе **BMAD METHOD**.

Инструмент спроектирован **исключительно для инженерии требований** и полностью очищен от любых процессов классической разработки ПО (написание кода, спринты, задачи разработчиков, Epics/Stories), унаследованных от базового БМАД-метода.

## 💡 Философия IDE-Native AI Agents

Requirements Mind v2.0 полностью интегрирован со средой разработки:
1. **Исполняемым движком ИИ является ваш собственный ИИ-помощник в IDE** (Cursor, Claude Code, Antigravity). У них уже есть доступ к лучшим языковым моделям и контексту. ИИ-агенты запускаются как локальные Markdown-навыки (skills) прямо из папки проекта.
2. **CLI-инструмент (`cli.py`) выступает в роли чистого контроллера файлов и состояний (Pure State & File Controller)**. Он не требует API-ключей, работает локально, проверяет переходы состояний в `state.json` и синхронизирует проект с Obsidian и NotebookLM.
3. **Накапливаемый источник истины (Cumulative Context Pattern):** Файл контекста `context.md` является стержнем проекта. Он защищен от перезаписи с нуля и сжатия. По мере прохождения цепочки BRD ➡️ SRS ➡️ Tech Design ИИ-агенты бережно встраивают новые Use Cases, сущности, структуры баз данных и API в соответствующие разделы, автоматически ведя реестр изменений в отдельном файле `context-changelog.md` рядом с ним.

## 📋 6 Целевых профилей проектов требований

При инициализации проекта система помогает выбрать один из **6 специализированных профилей требований**, в зависимости от целей и масштаба вашего проекта:

1. **Профиль 1: «Бизнес-концепт»** (цель: концептуальные бизнес-требования **BRD**).
2. **Профиль 2: «Системная спецификация»** (цель: детальные требования и логика системы **BRD ➡️ SRS**).
3. **Профиль 3: «Архитектурное проектирование»** (цель: архитектурные потоки и структура БД **BRD ➡️ SRS ➡️ Tech Design**).
4. **Профиль 4: «Интеграция и API»** (цель: полный цикл проектирования, включая REST/gRPC контракты **BRD ➡️ SRS ➡️ Tech Design ➡️ API Contract**).
5. **Профиль 5: «Аналитическое исследование (Режим B)»** (цель: свободный анализ требований, сравнения документов в папке **`analysis/`**).
6. **Профиль 6: «Сбор требований» (Elicitation Mode)** (цель: интерактивный сбор требований с нуля через ИИ-интервью в чате, если у аналитика есть только идея и нет сырых входов).

## 💬 Режим «Сбор требований» (Elicitation Mode)

Режим сбора требований с нуля (Профиль 6) создан для решения проблемы «чистого листа». Если у вас есть только сырая идея проекта, ИИ-интервьюер проводит с вами глубокий диалог по гибридной методологии **«Практичный аналитик»**:

* **Бизнес-ценность и JTBD (Jobs-to-be-Done):** выявление ключевых проблем, пользователей, их мотивации и конечной бизнес-ценности по формуле *«Когда я..., я хочу..., чтобы...»*.
* **Границы и сценарии (Use Cases):** описание ролей пользователей, их взаимодействия с системой и функциональных блоков.
* **Системные требования (NFR по ISO/IEC/IEEE 29148):** выявление критических нефункциональных требований (производительность, масштабируемость, безопасность, надежность, интеграция).

По окончании диалога ИИ-интервьюер **самостоятельно** генерирует и сохраняет на диск файл **`projects/<project-name>/input/requirements.md`**, который становится полноценным фундаментом для цепочки документов (BRD, SRS и т.д.).

## 🗣️ Уникальные коды команд ИИ (Capabilities Menu) в чате IDE

Чтобы не писать ИИ-помощнику в чате длинные промпты руками, мы полностью перевели меню агентов БМАД (Мэри, Джон, Пейдж) на **уникальные, непересекающиеся требования-ориентированные коды**. 

> [!IMPORTANT]
> Все стандартные программные коды БМАД (такие как `DP`, `BP`, `MR`, `DR`, `TR`, `CB`, `WB` и др.) **полностью заблокированы и заглушены** для предотвращения галлюцинаций ИИ-моделей в Cursor или Claude Code. Используйте исключительно новые уникальные `RM...` коды:

| Код команды | Название навыка | Что делает ИИ-агент |
| :---: | :--- | :--- |
| **`RMONB`** | **Онбординг проекта** (`a0-onboarding-wizard`) | Запускает Project Discovery, опрашивает по 6 профилям требований и выдает пошаговый Roadmap. |
| **`RME`** | **Сбор требований с нуля** (`a3-elicitation`) | Проводит сократовское интервью по методологии «Практичный аналитик» и создает `requirements.md`. |
| **`RMIN`** | **Intake-анализ сырых данных** (`a1-intake-analyst`) | Анализирует файлы в `input/`, убирает шум и создает структурированный `context.md` (без сжатия данных). |
| **`RMQ`** | **Генерация уточняющих вопросов** (`a3-question-generator`) | Проводит интерактивный опрос прямо в чате IDE (с поддержкой свободного ответа) для закрытия пробелов в контексте. |
| **`RMDW`** | **Написание черновика** (`a4-document-writer`) | Генерирует первый черновик спецификации (BRD, SRS, Tech Design, API) строго по чеклистам базы `kb/`. |
| **`RMVAL`** | **Жесткая валидация** (`a2-requirements-validator`) | Проверяет качество черновика на edge-cases и полноту, выдает отчет с вердиктом `PASSED`/`FAILED`. |
| **`RMAN`** | **Аналитическое исследование** (`a6-analysis-writer`) | Проводит свободный анализ (риски, противоречия, сравнение документов) в папку `analysis/`. |
| **`RMAUG`** | **Доработка существующего документа** (Augment Mode) | Аккуратно дополняет уже существующий SRS/BRD/Tech-Design новыми артефактами, **сохраняя структуру и формулировки** baseline. Обязательный diff-план в чате + подтверждение пользователя до записи. Подробно: [`docs/walkthroughs/augment-dams-srs.md`](docs/walkthroughs/augment-dams-srs.md). |
| **`reqmind`** | **Меню управления** (`reqmind`) | Интерактивный навигатор Requirements Mind. Выводит список всех RM-кодов, команд CLI и сценариев. |

> [!TIP]
> **Супер-команда `/reqmind`:** Наберите в чате вашей IDE **`/reqmind`** (в Claude Code / Antigravity) или упомяните **`@reqmind`** (в Cursor), и ИИ-ассистент мгновенно выведет интерактивное Capabilities Menu со всеми доступными агентами, CLI-командами и сценариями работы! Вы также можете использовать и запускать любые RM-коды по отдельности.

### 🗣️ Как вводить команды в чате IDE (3 формата)

ИИ-агенты Requirements Mind очень гибки и распознают команды в любом удобном для вас формате:
1. **Как обычное слово в предложении (Рекомендуется):** ИИ автоматически вычленит уникальный код из вашего текстового запроса.
   * *«Мэри, привет! Запусти RME для нового проекта delivery-app»*
   * *«Джон, сделай RMDW для проекта my-app, нужен черновик SRS»*
2. **Как короткую «команду-выстрел»:** Отправьте в чат просто код и название проекта:
   * `RMONB my-project`
   * `RME delivery-app`
3. **Через слэш (`/`):** Если вы привыкли к Slash-интерфейсам:
   * `/RMONB my-project`
   * `/RMIN my-project`

### 🔍 Как интегрировать RM-коды в выпадающее меню `/` и `@` вашей IDE

Поскольку разные IDE устроены по-разному, возможности настройки выпадающего списка при вводе слэша (`/`) различаются:

#### 1. В IDE Cursor (через механизм `@` At-Mentions)
В Cursor выпадающий список по клавише `/` (выводящий `/ask`, `/edit`, `/code` и др.) жестко зашит на уровне исходного кода самой IDE, и в настоящее время Cursor не предоставляет открытого API для добавления кастомных команд в это конкретное меню.

**Но у Cursor есть гораздо более мощный нативный инструмент — меню `@` (At-Mentions):**
* Введите в строке чата символ **`@`** и начните писать имя навыка, например: **`@a0-onboarding-wizard`** или **`@a3-elicitation`**.
* Cursor выведет подсказку и позволит прикрепить файл навыка прямо к вашему сообщению.
* После этого напишите любой текст (например: *«Запусти для my-app»*), и ИИ мгновенно применит этот навык к диалогу с 100% точностью!

#### 2. В Google Antigravity SDK и Claude Code (Автоматически)
В этих средах выпадающий список по `/` строится динамически на основе метаданных навыков (skills). 
Поскольку наш установщик `install.py` скопировал все md-файлы навыков с YAML-заголовками в папки `.agent/skills/` и `.claude/skills/`, ИИ-ассистенты **автоматически регистрируют** их в своем внутреннем списке команд. При вводе `/` они будут выводиться в выпадающем меню с нашими описаниями!

#### 3. В OpenAI Codex CLI (Директивы в терминале)
Скрипт автоустановки копирует все навыки в папку `.codex/skills/`. Вы можете вызывать агентов прямо из терминала:
```bash
codex --model=gpt-4o --prompt="@AGENTS.md @.cursorrules Запусти RMONB для проекта <name>"
```
*Важно явно указывать файлы `@AGENTS.md` и `@.cursorrules` в вашем промпте, чтобы предоставить агенту полный контекст правил команды и Capabilities Menu.*

## ⚙️ Быстрый старт (Quick Start)

### 1. Установка окружения с нуля в 1 команду
Чтобы развернуть и настроить Requirements Mind с нуля в любой папке на вашем компьютере, выполните в терминале одну команду — выберите блок под вашу ОС:

<details open>
<summary><b>🍎 macOS / 🐧 Linux (Bash / Zsh)</b></summary>

```bash
curl -fsSL https://raw.githubusercontent.com/Menta1ik/requirements-mind/main/install.py -o install.py && python3 install.py
```

</details>

<details>
<summary><b>🪟 Windows (PowerShell)</b></summary>

```powershell
Invoke-WebRequest -Uri https://raw.githubusercontent.com/Menta1ik/requirements-mind/main/install.py -OutFile install.py; python install.py
```

> **Требуется:** Python 3.10+ (`python --version`), Git и PowerShell 5.1+ (встроен в Windows 10/11).
> Если `python` не находится — установите Python с [python.org](https://www.python.org/downloads/) и отметьте чек-бокс **«Add Python to PATH»**.

</details>

*Эта команда автоматически скачает установщик, загрузит дистрибутив ядра с GitHub, настроит виртуальное окружение Python, установит все зависимости, свяжет ИИ-навыки с выбранными IDE и подготовит визуальный Obsidian Vault с демо-проектом. Установщик сам определяет ОС и подставляет правильные пути venv (`bin/` для Unix, `Scripts/` для Windows).*

### 2. Автономный импорт из Web/Confluence в 1 клик (Новинка!)
Если исходные материалы или требования находятся в закрытом корпоративном Confluence, Jira или на веб-странице, к которой у вас есть доступ в браузере:

1. **Запустите Chrome с портом отладки** — выберите блок под вашу ОС:

   <details open>
   <summary><b>🍎 macOS</b></summary>

   ```bash
   open -a "Google Chrome" --args --remote-debugging-port=9222
   ```

   </details>

   <details>
   <summary><b>🐧 Linux</b></summary>

   ```bash
   google-chrome --remote-debugging-port=9222
   # или: chromium --remote-debugging-port=9222
   ```

   </details>

   <details>
   <summary><b>🪟 Windows (PowerShell)</b></summary>

   ```powershell
   Start-Process "chrome.exe" -ArgumentList "--remote-debugging-port=9222"
   ```

   > Если `chrome.exe` не находится в PATH — укажите полный путь, например:
   > `Start-Process "C:\Program Files\Google\Chrome\Application\chrome.exe" -ArgumentList "--remote-debugging-port=9222"`

   </details>

   *(И откройте нужную вкладку со статьей Confluence)*.
2. **Попросите ИИ-агента в чате IDE:**
   > **«Мэри, импортируй требования из вкладки Confluence для проекта my-project»**
3. **ИИ-агент самостоятельно предложит и запустит команду импорта:**
   ```bash
   uv run cli.py import-web --project=my-project --port=9222 --query="confluence" --filename="confluence_specs.md"
   ```
   *(Вам нужно просто нажать кнопку подтверждения запуска в чате IDE. Скрипт сам подключится к Chrome по CDP, выкачает статью, переведет в Markdown и сохранит в `input/`)*.

> [!NOTE]
> **Быстродействие и переносимость (CDP-подключение):**
> Благодаря использованию протокола CDP (Chrome DevTools Protocol) скрипт импорта подключается к вашему **уже запущенному браузеру**. Это означает, что **установка тяжелых бинарников браузеров Playwright (`playwright install`) НЕ требуется**. Мы используем ваш собственный Chrome, его куки и активную авторизацию, что делает установку мгновенной, а работу — безопасной.



### 3. Пример полного конвейера (Сбор требований с нуля)
Разработка требований полностью автоматизирована и ведется прямо через диалог с ИИ в вашей IDE. Все необходимые команды CLI предлагаются и запускаются ИИ-агентом автоматически:

1. **Интервью и инициализация (RME):**
   * Напишите Мэри в чате: **`«Запусти RME для нового проекта my-delivery-app»`**.
   * ИИ-агент **самостоятельно создаст структуру папок**, сгенерирует `state.json`, проведет с вами интервью по методологии «Практичный аналитик» и запишет требования в `input/requirements.md` и `context.md`.
   * ИИ-агент автоматически предложит запустить онбординг:
     ```bash
     # ЗАПУСКАЕТСЯ АВТОМАТИЧЕСКИ ИИ-АГЕНТОМ ПОСЛЕ ПОДТВЕРЖДЕНИЯ В ЧАТЕ
     uv run cli.py onboard --project=my-delivery-app
     ```
2. **Разбор требований и закрытие пробелов (RMIN):**
   * Напишите ИИ-агенту: **`«Запусти RMIN для my-delivery-app»`**.
   * ИИ проанализирует контекст на полноту. Если найдутся пробелы, он проведет опрос с возможностью свободного ввода ответов.
   * После устранения пробелов ИИ-агент автоматически предложит выполнить команду фиксации разбора (Intake):
     ```bash
     # ЗАПУСКАЕТСЯ АВТОМАТИЧЕСКИ ИИ-АГЕНТОМ ПОСЛЕ ПОДТВЕРЖДЕНИЯ В ЧАТЕ
     uv run cli.py intake --project=my-delivery-app
     ```
3. **Генерация черновика спецификации (RMDW):**
   * Напишите ИИ-агенту: **`«Запусти RMDW для my-delivery-app, нужен черновик BRD»`**.
   * ИИ создаст черновик документа `BRD-v1.md` в папке `draft/` строго по чеклистам базы знаний `kb/`.
   * ИИ-агент автоматически предложит зафиксировать черновик:
     ```bash
     # ЗАПУСКАЕТСЯ АВТОМАТИЧЕСКИ ИИ-АГЕНТОМ ПОСЛЕ ПОДТВЕРЖДЕНИЯ В ЧАТЕ
     uv run cli.py draft --project=my-delivery-app --doc=BRD
     ```
4. **Жесткий контроль качества (RMVAL + Tier 1 trace):**
   * Перед запуском LLM-валидатора рекомендуется прогнать дешёвую детерминированную проверку формы — линтер ID и трассируемости:
     ```bash
     uv run cli.py trace --project=my-delivery-app
     ```
     Он ловит дубли `FR-01`, orphan-ссылки (ID упомянут, но нигде не определён), Business Goals без покрытия FR в SRS — за миллисекунды, без LLM. Подходит для pre-commit / CI.
   * После — запускайте семантическую LLM-валидацию: **`«Запусти RMVAL для my-delivery-app»`**.
   * ИИ-агент A2 проверит черновик на логические нестыковки, пограничные случаи (edge cases) и упущенные детали, сформирует отчет с вердиктом `PASSED`/`FAILED`.
   * ИИ-агент автоматически предложит зафиксировать результаты валидации:
     ```bash
     # ЗАПУСКАЕТСЯ АВТОМАТИЧЕСКИ ИИ-АГЕНТОМ ПОСЛЕ ПОДТВЕРЖДЕНИЯ В ЧАТЕ
     uv run cli.py validate --project=my-delivery-app --doc=BRD --version=1
     ```
5. **Финализация и переход на следующий этап (Final):**
   * Если валидация пройдена (`PASSED`), ИИ-агент автоматически предложит финализировать документ:
     ```bash
     # ЗАПУСКАЕТСЯ АВТОМАТИЧЕСКИ ИИ-АГЕНТОМ ПОСЛЕ ПОДТВЕРЖДЕНИЯ В ЧАТЕ
     uv run cli.py final --project=my-delivery-app --doc=BRD --version=1
     ```
   * Документ будет скопирован в `final/`, а проект перейдет на этап проектирования системной спецификации (`SRS`).

## 🔄 Обновление

Чтобы обновить ядро Requirements Mind, зависимости и ИИ-навыки во всех средах разработки (Cursor, Claude Code, Antigravity, OpenAI Codex) до актуальной версии с GitHub, выполните одну команду в терминале — она одинакова на всех ОС:

```bash
uv run cli.py update
```

> **Если у вас не установлен `uv`** — используйте Python из вашего venv напрямую:
>
> <details>
> <summary><b>🍎 macOS / 🐧 Linux</b></summary>
>
> ```bash
> .venv/bin/python cli.py update
> ```
>
> </details>
>
> <details>
> <summary><b>🪟 Windows (PowerShell)</b></summary>
>
> ```powershell
> .\.venv\Scripts\python.exe cli.py update
> ```
>
> </details>

**Что делает команда:**

1. **Обновление файлов ядра:**
   * Если проект — Git-репозиторий → выполняется `git pull origin main`.
   * Если проект развёрнут из ZIP-архива → скачивается свежий ZIP с GitHub и бережно обновляются только файлы ядра: `cli.py`, `install.py`, `requirements.txt`, `skills/`, `kb/`, `docs/`, `flows/`, `scripts/`.
2. **Обновление зависимостей:** проверяются и доустанавливаются новые библиотеки из `requirements.txt` (через `uv` или `pip`).
3. **Обновление ИИ-навыков IDE:** автоматически перезапускается `setup-ide`, копируя свежие навыки и `.cursorrules` во все ваши ИИ-ассистенты (Cursor, Claude Code, Google Antigravity, OpenAI Codex).

> [!IMPORTANT]
> **Что гарантированно НЕ трогается при обновлении:**
> * `projects/` — все ваши рабочие проекты;
> * `.env` и `.env.local` — ваши локальные настройки;
> * `vault/` и `notebooklm/` — сгенерированные базы и экспорты;
> * любые ваши кастомизации вне списка файлов ядра.

📖 Полное описание процесса обновления и сценариев восстановления — в [`docs/user_guide.md` §5](docs/user_guide.md#5-automatic-requirements-mind-update-one-click-update).

## 🩺 Диагностика и ручное управление FSM

Если автоматика ИИ-агента дала сбой (не услышал RM-код, неправильно сформировал артефакт, IDE отказалась запускать CLI), у вас есть две полностью ручные команды для отладки и отката.

### `cli.py status` — что сейчас происходит с проектом

```bash
uv run cli.py status --project=my-app
```

Read-only обзор: текущий статус FSM, документ/итерация, активные агенты, открытые вопросы, маркер `rm_status` в `context.md` и сводка артефактов по всем шести поддиректориям проекта. Ничего не пишет, безопасно вызывать в любой момент.

### `cli.py reset` — принудительный перевод FSM

```bash
# С подтверждением в TTY
uv run cli.py reset --project=my-app --to=intake

# Без подтверждения (для скриптов)
uv run cli.py reset --project=my-app --to=drafting --yes
```

Разрешённые значения `--to`: `onboarding | intake | needs_questions | drafting | validating | needs_revision | approved`. Команда правит **только** `state.json` и автоматически выставляет `active_agents` под целевой статус. Артефакты на диске (`draft/`, `messages/`, `final/`) не трогаются.

### `cli.py trace` — Tier 1 детерминированная валидация ID

<p align="center">
  <img src="docs/screenshots/cli-trace.svg" alt="cli.py trace — Tier 1 deterministic линтер ID и трассируемости" width="900"/>
  <br/>
  <em>Команда <code>cli.py trace</code> — детерминированный regex-линтер ID. Ловит дубли (RM-TR-001), orphan-ссылки (RM-TR-002) и непокрытые Business Goals (RM-TR-020) за миллисекунды, без LLM.</em>
</p>

```bash
# Весь проект
uv run cli.py trace --project=my-app

# Один документ конкретной версии
uv run cli.py trace --project=my-app --doc=SRS --version=1

# Произвольный .md файл (вне проектной структуры)
uv run cli.py trace --file=path/to/SRS-v1.md
```

Регексп-линтер без LLM. **Дополняет** `RMVAL` (семантический LLM-валидатор A2), а не заменяет:

| Слой | Кто проверяет | Что ловит |
| :---: | :--- | :--- |
| **Tier 1** — `cli.py trace` | regex, миллисекунды | Дубли `FR-01` в одном документе, orphan-ссылки (ID упомянут, но нигде не определён), Business Goals из BRD без покрытия FR в SRS, форма ID (`FR-01`, `FR-REG-01`, `NFR-CAP-02`, `BG-1`) |
| **Tier 2** — `RMVAL` / `cli.py validate` | A2 (LLM), секунды-минуты | Семантика: измеримы ли NFR, есть ли альтернативы в Use Case, согласуется ли SRS с BRD, нет ли поверхностных формулировок |

**Exit codes** машиночитаемы (`0` — чисто, `1` — есть ошибки, `2` — ошибка вызова) — встраивается в pre-commit hook или CI без обвязки. **`state.json` не трогает** — это чистая проверка, не FSM-переход.

Канонический алфавит ID: 2-5 заглавных букв + 0-3 субдомена + 1-4 цифры. Примеры из «зелёной» зоны: `FR-01`, `FR-REG-01`, `NFR-CAP-02`, `UC-REG-AS-1`, `VAL-03`. Не пройдёт: `fr-01` (строчные), `FR_01` (подчёркивание), `FR-01234567` (хвост >4 цифр — warning `RM-TR-010`).

## ⚠️ Ограничения и известные риски (Limitations)

Requirements Mind — это «детерминированный CLI-контроллер поверх недетерминированных ИИ-агентов». Чтобы избежать завышенных ожиданий, ниже честно перечислены границы возможностей текущей версии.

### Поддержка IDE

| IDE                          | Уровень поддержки         | Что работает / что нет                                                                                                                     |
| :---                         | :---                      | :---                                                                                                                                       |
| **Claude Code**              | ✅ Протестировано           | Слэш-команды (`/RMONB`, `/reqmind`, …) подхватываются из `.claude/skills/`. Автозапуск CLI требует подтверждения в чате.                |
| **Cursor**                   | ✅ Протестировано           | Меню `/` нативно не расширяется (закрытая часть Cursor), но `@`-меню видит скиллы из `.agents/skills/`. RM-коды распознаются в свободном тексте. |
| **Google Antigravity**       | 🟡 Best-effort             | Подхват скиллов из `.agent/skills/` ожидаем по тем же правилам, что у Claude Code, но end-to-end сценарии тестируются по факту.            |
| **OpenAI Codex CLI**         | 🟡 Best-effort             | Скиллы копируются в `.codex/skills/`. Требуется явное упоминание `@AGENTS.md @.cursorrules` в каждом промпте для воспроизводимого поведения.  |

### Чего инструмент не гарантирует

1. **ИИ-агент может «не услышать» RM-код.** Особенно при коротких запросах, очень длинной истории чата или конкурирующих инструкциях системного промпта IDE. **Что делать:** повторите команду явным форматом — `RMIN <project-name>` отдельным сообщением, либо запустите CLI-этап руками (см. ниже).
2. **ИИ может неправильно сформировать `context.md` / отчёт валидации.** В частности — забыть управляющий frontmatter (`rm_status`, `rm_verdict`). CLI **не сломается**: для этих случаев есть устаревший substring-fallback, который выводит warning и просит агента добавить frontmatter. Но решение FSM в этом режиме менее надёжно — проверяйте.
3. **Автозапуск CLI из чата — не магия.** ИИ-агент *предложит* команду, но фактически выполнить её в терминале вашей IDE — это её собственная фича (Cursor Composer, Claude Code shell-вызовы и т. д.). Если в вашем окружении она недоступна, скопируйте и запустите команду руками.
4. **Конкурентная работа двух аналитиков с одним проектом не поддержана.** `state.json` правится без блокировок. Для совместной работы используйте Git и не запускайте `cli.py` параллельно на одной папке проекта.
5. **`projects/<name>/` — не source of truth.** Это рабочая папка одного аналитика. Постоянное хранилище — Git и/или Obsidian Vault.

### Аварийный ручной режим (если автоматика молчит)

Любой этап конвейера выполняется вручную через CLI, минуя чат:

```bash
uv run cli.py intake   --project=my-app
uv run cli.py draft    --project=my-app --doc=BRD
uv run cli.py validate --project=my-app --doc=BRD --version=1
uv run cli.py final    --project=my-app --doc=BRD --version=1
```

Для отладки застрявшего FSM смотрите раздел [«Диагностика и ручное управление FSM»](#-диагностика-и-ручное-управление-fsm) — там описаны команды `status` и `reset`.

### Тесты

CLI покрыт pytest-набором (`tests/`): FSM-переходы intake/validate (включая обе ветки frontmatter и fallback), парсер frontmatter, санитизация имени проекта, поведение `reset`. Запуск:

```bash
uv run pytest
```

Покрытие — именно слой контроллера состояний (то, что реально может сломаться при правках `cli.py`). Поведение самих ИИ-агентов pytest'ом не покрывается — это вопрос модели и контекста IDE, а не нашего кода.
